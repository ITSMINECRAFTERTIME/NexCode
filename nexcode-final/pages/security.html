Foundations

Three pillars of protection

🧬

Virtualization

Your code never runs as recognizable Lua. It's transformed into a custom virtual machine with a unique instruction set generated fresh for every compile.

🔐

Encryption

Strings, constants, and function bodies are encrypted at rest and only decrypted in memory at the moment of execution — never written to disk in plaintext.

🛡️

Runtime defense

Anti-tamper hooks continuously verify the environment, detecting debuggers, memory hooks, and instrumentation — and can trigger a kill switch.

Threat model

Attacks we anticipate

Security is only meaningful against real attacks. Here's what adversaries try — and how NexCode answers.

🔍

Automated decompilation

Attacker runs unluac or similar to recover readable source.

Anti-decompiler injection crashes or hangs these tools; virtualized code has no Lua source to recover.

🐛

Live debugging

Attacker attaches a debugger to step through execution.

Anti-tamper hooks detect debuggers and hooks, and can halt or corrupt execution on detection.

📋

Key sharing / reselling

A buyer shares or resells their key to others.

HWID locking binds keys to hardware; resets are logged and abuse patterns surface in the audit trail.

🌐

Endpoint extraction

Attacker sniffs traffic to steal webhook or API URLs.

Encrypted webhook proxying keeps real endpoints out of the distributed script entirely.

♻️

Build diffing

Attacker compares two builds to isolate logic.

Every compile randomizes the VM, opcodes, and layout — builds share no reusable structure.

🎣

Memory dumping

Attacker dumps memory to catch decrypted strings.

Strings decrypt only at point of use and are short-lived; constant obfuscation hides values in transit.

HWID protection

How hardware locking works

1

First execution

On first run, NexCode collects a privacy-respecting hardware fingerprint and binds it to the key.

2

Every subsequent run

The auth check compares the current fingerprint against the bound one in under 50ms.

3

Mismatch handling

If the hardware doesn't match, the key fails silently — no useful feedback for someone probing it.

4

Controlled resets

Users can reset their own HWID within limits; managers can force-reset. Every reset is logged for abuse detection.
