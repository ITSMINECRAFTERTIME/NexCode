No commercially available tool can automatically devirtualize NexCode output. The VM randomizes opcode mappings on every compile, so reverse-engineering one build yields nothing reusable for the next. Anti-decompiler injection additionally crashes or hangs standard tools like unluac.
It can add overhead, which is why we give you control. Use NXC_NO_VIRT on hot paths like render loops and physics steps to skip the VM there, and pick the Balanced or Fast optimization profile. Most users report no perceptible difference with the right settings.
Fast applies lighter protection with minimal performance cost. Balanced (recommended) gives strong protection with reasonable overhead. Max applies every layer at full strength for high-value scripts where security matters more than speed.

On first execution, NexCode captures a hardware fingerprint and binds it to the key. Every subsequent auth check validates the machine matches. Shared keys fail silently on non-matching hardware. Admins can reset fingerprints via /resethwid or the REST API, and every reset is logged.
